NoteFragments ドキュメント仕様書

1. 開発環境
開発確認は Vivaldi で行う。
利用者の前提ブラウザは Chrome とする。
実装は Chromium 系ブラウザを基準に考える。
File System Access API を使える前提で設計を進める。
Firefox 依存の挙動は初期版の前提にしない。
必要に応じて、後から Firefox 向けの代替案を別仕様として扱う。

1.1 動作確認
動作確認は、Vivaldi 上で以下を確認できることを目標とする。
- 拡張機能の再読み込み
- テキスト、URL、画像の保存
- ドキュメントの編集
- Markdown export
- `NFAssets` を伴うローカル出力
- サムネイル、圧縮版、完全版の画像表示

1.2 確認基準
確認は、UI が表示されることだけではなく、保存した内容が再起動後に再取得できることを基準とする。
export は、実際にローカルファイルが生成され、オフラインで開けることを基準とする。


1. 概要
NoteFragments は、テキスト、URL、画像、ドキュメントをブラウザ内で収集・閲覧・編集するための機能である。
まず軽いメモとして保存し、必要に応じて構造化されたドキュメントへまとめることを基本方針とする。
開発確認は Vivaldi、利用者前提は Chrome とする。
native messaging は標準機能には含めず、将来必要になった場合にのみ拡張機能として検討する。


2. 目的
1. テキスト、URL、画像を後から見返しやすい形で保存する。
2. 軽い束ね操作と正式なドキュメント化を分ける。
3. ブラウザ内で閲覧と編集を完結できるようにする。
4. ローカルファイルへの export を可能にする。
5. 画像を含む大量データでも扱いやすいようにする。
6. 将来的に Eagle のようなコレクション管理へ発展できる余地を残す。


3. 基本概念
3.1 フラグメント
フラグメントとは、NoteFragments で扱う最小単位である。
対応するフラグメント種別は以下のとおり。
- TEXT
- URL
- IMAGE

3.2 ドキュメント
ドキュメントとは、複数のフラグメントを組み合わせた構造化単位である。
閲覧、編集、export の対象となる正式な保存単位として扱う。

3.3 軽い束ね
軽い束ねとは、フラグメント同士を一時的または半正式に関連付ける操作である。
正式なドキュメントを作らなくても、後から整理しやすい形にまとめるために使う。

3.4 コレクション
コレクションとは、テキスト、URL、画像、ドキュメントをまとめて扱う将来の概念である。
初期版では必須ではない。


4. 対応するフラグメント種別
4.1 TEXT
- ただの文字列として扱う。
- 通常のメモ、短文、説明文、キャプションに使う。
- 最も軽い基本フラグメントである。

4.2 URL
- URL 文字列と、必要に応じてタイトルを保持する。
- 画面上ではリンクとして扱う。
- 機能としては TEXT に近いが、表示と操作を分ける。

4.3 IMAGE
画像は次の3段階で扱う。
- サムネイル
- 圧縮版
- 完全版

画像メタデータは以下を持つ。
- sourceUrl: 元ページや出典のURL
- imageUrl: 元画像の参照先
- thumbnailUrl: 一覧用の縮小画像
- alt: 代替テキスト
- caption: 補足説明

画像の扱いに関するルール。
- 画像には、可能な限り出典情報を持たせる。
- フラグメント一覧や一覧ではサムネイルのみを表示する。
- 詳細表示では圧縮版を優先する。
- 完全版は export 時やローカル保存時に使う。
- メモとして保持する画像の数は、UI が重くならないよう制限してよい。


5. ドキュメント構造
5.1 構成
ドキュメントは TEXT、URL、IMAGE の組み合わせから成る。
各フラグメントはドラッグ＆ドロップで配置できる。
ドキュメントは、複数の関連フラグメントを正式にまとめたものとして扱う。

5.2 編集
- ドキュメントは閲覧モードと編集モードを持つ。
- 編集モードでは、フラグメントの追加、削除、並べ替えを行える。
- 編集結果は保存前に確認できる。
- ただの文字列編集だけでなく、構造を意識した編集UIを用意する。

5.3 閲覧
- ドキュメントのプレビュー欄は折りたたみ・展開ができる。
- export 時やプレビュー表示時には、Markdown 的に読みやすい見た目を優先する。
- 画像は raw データ文字列ではなく、視覚要素として表示する。


6. 保存方針
6.1 ブラウザ内保存
browser.storage.local は以下に使う。
- アプリ設定
- UI 状態
- ブラウザ再起動後も残したい軽量データ
- 軽い索引やキャッシュ

6.2 正本の考え方
データの正本は、機能ごとに明確にする。
推奨方針は以下のとおり。
- ブラウザ内保存: 運用中のデータと設定
- ローカル export: オフライン用の保存成果物
- export は、明示的な同期機能を追加しない限り、ライブ同期先とはみなさない

6.3 永続性
- メモ、URL、ドキュメント状態はブラウザ再起動後も残るべきである。
- 拡張機能のデータを消さない限り、通常は失われない。
- 拡張機能を削除したり、ブラウザデータを消去した場合は失われる。


7. 画像保存方針
7.1 ブラウザ内での画像扱い
- 一覧やタイムラインではサムネイルを使う。
- 詳細表示では圧縮版を使う。
- 表示が重い場合は遅延読み込みを使う。

7.2 ローカル export 時の画像扱い
- 完全版の画像をローカルファイルとして保存する。
- Markdown export の場合、画像は `NFAssets` フォルダへ出力する。
- Markdown の画像参照は相対パスにする。

7.3 export 例
ExportRoot/
  document.md
  NFAssets/
    image-001.jpg
    image-002.png

export では、ドキュメント本体と画像ファイルの対応関係を維持する。

7.4 画像圧縮
- 必要に応じて、ブラウザ内で圧縮版を生成または保持する。
- 圧縮するかどうかは設定で切り替えられるようにする。
- 圧縮は、表示速度と保存容量のバランスを取るために使う。


8. export 方針
8.1 対応形式
対応する export 形式は以下とする。
- .md
- .txt

8.2 Markdown export
- Markdown export を基本とする。
- 画像がある場合は、本文 .md と `NFAssets` フォルダをセットで出力する。
- Markdown 内の画像参照は相対パスで記述する。
- オフラインでもフォルダ単位で開けば閲覧できることを目指す。

8.3 テキスト export
- テキスト export は簡易出力とする。
- 基本的には TEXT と URL を中心に出力する。
- IMAGE は原則として含めない。

8.4 export 実行
- export は手動で行う。
- 自動 export は初期機能としては必須にしない。
- 互換性のため、export はブラウザ標準機能だけで使える形を優先する。

8.5 保存ダイアログ
- Firefox では、保存ダイアログの初期パスを自由に制御できる前提にしない。
- ただし、アドオン側で前回使った保存先を覚えて UI に表示することはできる。
- 保存ダイアログの起点位置そのものを保証する仕様にはしない。


9. 設定
アドオンが保持する設定は以下とする。
- ドキュメントのデフォルト保存先
- 画像のデフォルト保存先
- 前回使った保存先の表示
- export 形式の既定値
- 画像を圧縮して保存するかどうか
- UI に関する基本設定

設定は UI 上で確認でき、ブラウザ再起動後も保持されるものとする。


10. サイドバー
10.1 一覧表示
- TEXT、URL、IMAGE を一覧できる。
- IMAGE はカード表示にして見やすくする。
- 保存済みのドキュメントを一覧から開けるようにする。

10.2 操作
- TEXT と URL はドラッグ＆ドロップでドキュメントへ追加できる。
- 複数の TEXT / URL をまとめて 1 つのドキュメントに合成できる。
- IMAGE も将来的に同じ操作体系へ含める。
- 軽い束ね操作をサイドバーから行えるようにする。


11. 閲覧・編集の流れ
11.1 開く
- 保存済みのデータ、または export 済みのファイルを指定して開ける。
- ブラウザ内データとローカルファイルの両方を扱えることを目指す。

11.2 閲覧
- 閲覧モードは読み取り専用とする。
- プレビューは折りたたみ・展開が可能であることが望ましい。
- 画像は視覚的に見やすい形で表示する。

11.3 編集
- 編集モードではフラグメントの内容と順序を変更できる。
- 保存前の変更は明示する。
- 保存するとブラウザ内のドキュメント状態が更新される。


12. ローカル出力の考え方
- ローカル出力は、ブラウザ内の保存とは別に扱う。
- export により、オフライン閲覧用のファイル一式を作る。
- 必要であれば、将来 native messaging を使ってローカル整理機能を追加できる。
- ただし初期版では、標準機能として native messaging を必須にしない。


13. 将来拡張
将来的に追加してよい機能は以下とする。
- メモ同士や URL 同士のより強い関連付け
- 画像を含むコレクション管理
- 検索・索引の強化
- メモ、URL、画像、ドキュメントの相互参照
- 必要になった場合のみ native messaging を使う拡張版


14. 初期版の非目標
初期版では、次の機能は必須としない。
- 自動ローカル同期
- native messaging の必須化
- ブラウザ非依存の任意パス制御
- Eagle 同等の完全な資産管理機能


15. 実装上の注意
- まずはブラウザ標準の機能で完結する形を優先する。
- 軽い束ねと正式なドキュメント化は分ける。
- 画像はサムネイルと圧縮版で軽く見せる。
- export をオフライン保存の基本導線とする。
- 将来拡張は optional にして、最初から重くしない。
